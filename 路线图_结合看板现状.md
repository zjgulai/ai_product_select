### 1.2 关键发现

**唯一真实工作的页面：数据管理（DataManager）**

这是整个项目的"隐藏宝藏"。它已经实现了：
- Excel/CSV文件上传 + 本地解析 + 预览
- 模板选择器（按page分组：tiktok/amazon/report）
- 数据入库（通过tRPC调用`bulkInsert`写入`dynamic_data`表）
- 数据管理（查看已入库数据集、导出Excel、删除数据）
- 文件上传记录管理

**这意味着**：后端已经有一个"万能数据注入器"。其他页面只需要把mock数据源切换到tRPC查询即可接入真实数据。

**前端代码重复最严重的区域**：Amazon趋势下的6个页面（Keyword/List/Product/ParamTrend/BrandTrend/HotMarket/PotMarket）结构几乎完全相同——搜索区 + Tab（部分有）+ CategoryFilter + 筛选条件 + 数据表格 + 导出按钮。每个页面200-300行，其中60%是重复代码。

**交互完成度最高的页面**：
1. TikTokProducts（商品）：有真实工作的前端过滤（搜索、价格区间、评分、Tab切换），有收藏功能，有Toast提示
2. AmazonKeyword（关键词）：搜索后切换视图（统计卡片 → 示例报告 → 搜索结果），有状态管理
3. ReportAnalysis（报告）：7个Tab切换、导出Modal、收藏/对比、评价标签云交互、评论分析Modal

### 4.4 建立前端请求层规范

**当前问题**：
- 有些页面用`useBatchDynamicData`（tRPC）
- 有些页面直接import mock数据
- 没有统一的Loading/Error处理

**解决方案**：

```typescript
// src/hooks/useDataTable.ts
export function useDataTable<T>({
  queryFn,
  pageSize = 20,
}: {
  queryFn: () => UseQueryResult<T[]>;
  pageSize?: number;
}) {
  const [page, setPage] = useState(1);
  const [filters, setFilters] = useState<Record<string, any>>({});
  
  const query = queryFn();
  
  const data = useMemo(() => {
    if (!query.data) return [];
    // 前端分页
    return query.data.slice((page - 1) * pageSize, page * pageSize);
  }, [query.data, page, pageSize]);
  
  return {
    data,
    isLoading: query.isLoading,
    isError: query.isError,
    error: query.error,
    page,
    setPage,
    total: query.data?.length ?? 0,
    filters,
    setFilters,
  };
}
```

**目标**：建立TikTok↔Amazon的产品映射关系，推出融合选品看板。

### 5.1 L1 关键词映射（Week 1-3）

**数据来源**：
- TikTok侧：视频Hashtag、视频文案、达人带货描述、评论区高频词
- Amazon侧：搜索关键词、商品标题、Bullet Points、Review高频词

**实现方案**：

```
api/services/fusion/
  keywordMapper.ts       -- 关键词映射核心服务
  embeddingService.ts    -- Embedding生成（调用OpenAI/BGE）
  mappingEngine.ts       -- 相似度计算 + 映射决策
```

**算法流程**：
1. 从TikTok视频描述中提取名词短语（spaCy NER + noun chunks）
2. 从Amazon标题/搜索词中提取关键词
3. 使用BGE-multilingual模型生成embedding向量
4. 计算余弦相似度，>0.85建立映射，0.7-0.85推人工确认
5. 已确认的映射存入`keyword_mappings`表

**前端展示**：在Amazon关键词趋势页增加"TikTok关联热度"列

### 6.2 AI报告生成

**当前状态**：ReportAnalysis页面有7个Tab的UI，但数据是mock的，导出按钮是假的。

**改造后**：
1. 用户输入关键词/类目
2. 系统自动采集Amazon Top 100商品 + 10万条评论
3. NLP Pipeline分析
4. LLM（GPT-4o/Claude）生成结构化报告：
   - 市场概况（数据摘要）
   - 痛点清单（Top 10）
   - 机会点清单（Top 10）
   - 竞品对比分析
   - 产品改进建议
   - 内容营销建议（TikTok角度）
5. 报告可导出为PDF（使用Puppeteer + html-pdf）

### 6.3 自然语言查询

**新增功能**：用户可以用自然语言提问。

```
用户输入："帮我找3个月内上架、月销量增长超过50%、
          评论4.5星以上、TikTok热度在上升的厨房小工具"
          
系统解析：
  - 上架时间: < 90天
  - 销量增长率: > 50%
  - 评分: >= 4.5
  - 类目: Kitchen
  - TikTok SHI趋势: 上升
  
系统返回：符合条件的商品列表 + 融合分析
```

**实现**：LLM做意图识别和参数提取 → 转换为结构化查询 → 查询数据库 → LLM生成自然语言回答。

## 第九部分：执行建议

### 9.1 第一周立即开始的工作

1. **数据库Schema扩展**：在现有3张表基础上，新增`amazon_products`、`amazon_reviews`、`tiktok_videos`、`keyword_mappings`表。不要删除现有表，DataManager功能依赖它们。

2. **Amazon爬虫MVP**：用Playwright写一个能采集单个ASIN商品信息和前10页评论的脚本。这是整个平台的数据之源。

3. **Amazon关键词页接入真实数据**：这是用户价值感知最强的页面之一。先把关键词搜索从mock切换到真实API（哪怕数据不多）。

4. **统一Loading状态**：给所有页面加一个统一的Loading骨架屏组件，用户至少知道"系统在加载"而不是"页面坏了"。

### 9.2 资源投入建议

### 9.3 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| Amazon反爬导致数据采集中断 | Phase 1延期 | 多IP代理池 + 请求频率控制 + 第三方API备用方案 |
| TikTok API限制 | TikTok数据不足 | 优先建设Amazon侧，TikTok用有限API+人工标注映射 |
| LLM API成本过高 | 运营成本失控 | 自建轻量BERT模型做基础NLP，LLM只做复杂推理；国内用DeepSeek替代 |
| 团队人手不足 | 进度延期 | Phase 1优先只做Amazon侧数据管道，TikTok延后2-3周 |
